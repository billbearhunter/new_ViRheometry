1. PROJECT OVERVIEW
===================
ViRheometry: estimate Herschel-Bulkley (HB) rheology parameters from video of
dam-break flow. Parameters: n (power-law index), η (viscosity), σ_y (yield stress).

Pipeline:
  Video → Calibration (flow distances) → MoE Surrogate → CMA-ES Optimization

Surrogate: Mixture-of-Experts (MoE)
  - GMM gating network (K=100 clusters, diag covariance) on 18-dim φ features
  - Per-cluster GP expert: ExactGP (Matérn 2.5 + Linear kernel) on 5D input
    → 8D output (flow distances at frames 1-8)
  - Input: (n, η, σ_y, W, H)  Output: (x_01, ..., x_08) flow distances in cm

Ground truth for validation:
  n=0.5297, η=33.28, σ_y=40.57 (Tonkatsu sauce)

     inferred: n=0.6537, η=18.27, σ_y=88.73
     GT:       n=0.5297, η=33.28, σ_y=40.57
     → not great, root cause = GP surrogate error in queried clusters

c) BO data collection ATTEMPT (failed — important lesson)
   - Collected 600+ points using max-variance acquisition per cluster
   - PROBLEM 1: BO points bypassed GMM gating → forced into target cluster
     → data from different flow regimes mixed into wrong expert → GP degraded
   - PROBLEM 2: max-variance selects cluster-box edge points with extreme
     flow distances (2-3x val range) → GP hyperparameters pulled away from
     useful region
   - RESULT: all 4 clusters got WORSE (MaxErr increased 50-225%)
   - SOLUTION: rolled back to pre-BO baseline, rewrote bo_collect.py

d) bo_collect.py REWRITE (committed, ready to use)
   - NEW --mode lhs (default): Latin Hypercube Sampling + GMM filter
   - Flow: sample in cluster box → MPM simulate → build φ → GMM.predict()
     → only keep points GMM assigns to target cluster with conf >= 0.6
     → bonus points go to their GMM-assigned cluster
   - OLD --mode bo: variance-based (legacy, not recommended)
   - This ensures data consistency with the MoE gating network

e) Bug fixes committed:
   - bo_collect.py merge logic: fill cluster_id/cluster_conf for new rows
   - optimize_2setups.py: auto-gen setup3 data dir

=================

a) GP surrogate accuracy — THE main bottleneck
   9 clusters have val MaxErr > 1.0 cm (out of 100):
     cluster  51: MaxErr=5.787 cm (WORST, only ~150 train points)
     cluster  52: MaxErr=3.887 cm
     cluster  16: MaxErr=2.719 cm
     cluster  64: MaxErr=2.592 cm
     cluster   5: MaxErr=2.106 cm
     cluster  46: MaxErr=2.078 cm
     cluster  76: MaxErr=1.473 cm
     cluster  88: MaxErr=1.369 cm
     cluster  71: MaxErr=1.080 cm

   21 clusters have MaxErr > 0.5 cm
   42 clusters have MaxErr < 0.1 cm (good)
   Median MaxErr across all clusters: 0.155 cm

b) CMA-ES optimization path hits clusters 3, 21, 61, 88 for GT params
   → these clusters' errors directly affect parameter recovery

c) Simulation can hang on extreme parameters (taichi kernel deadlock)
   → observed during BO on macOS Metal; may not occur on CUDA

d) sklearn version mismatch warning when loading gmm_gate.joblib
   (saved with 1.7.2, loaded with 1.6.1) — functionally OK but verify on
   your Windows environment. If it errors, retrain GMM:
     python DataPipeline/prepare_data.py --data Optimization/moe_workspace5 --force

e) Known code issues (intentionally unfixed):
   - is_inner_of_box reset bug in Simulation
   - h=0.05 hardcode vs py_particle_hl=0.0315 mismatch
